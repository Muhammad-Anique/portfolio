A
Messaging Architecture — My Recommendation

Short answer: Option B — Evolution API only. No hybrid needed at your scale.

Here is my reasoning:

1. At 300 patients/month, the official Cloud API adds cost and complexity without clear benefit. The official API charges per conversation, requires template pre-approval (24-72h turnaround), and limits free-form messaging to a 24-hour window after the patient last messaged you. That window restriction directly conflicts with Sofia's need to send proactive follow-ups, D-15/D-7 reminders, and voucher messages at any time.

2. Evolution API (Baileys) already gives you everything you need. Free-form messages, buttons, lists, interactive flows, no template approval delays, no 24h window restriction. Sofia can send any message at any time. This is exactly what your conversational AI needs.

3. The dual-layer approach (Option C) is overengineering for your volume. Running two messaging channels means maintaining two webhook receivers, two payload formats, two delivery tracking systems, and routing logic to decide which channel handles which message. This doubles the WF_INGEST and WF_RESPOND complexity for no practical gain at 300 patients/month.

4. Latency consideration: Evolution API adds ~200-400ms over the official API. At your volume, this is imperceptible to patients. The latency concern only matters at 5,000+ concurrent sessions.

5. Future-proofing: If you ever need to migrate to the official Cloud API (for compliance or scaling beyond 2,000 patients/month), the architecture I am building uses an abstraction layer in WF_RESPOND. Switching the transport layer underneath is a 2-day effort, not a rewrite. So there is zero risk in starting with Evolution API only.
